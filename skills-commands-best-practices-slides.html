PART 1

SKILLS
VS COMMANDS

Two different extension points — same invocation syntax, very different scope and lifecycle.

01
COMMANDS

SLASH COMMANDS

WHAT IT IS

  • A single .md file dropped in .claude/commands/
  • Invoked with /filename (no extension) in any session
  • The file content becomes the prompt — nothing more
  • Supports $ARGUMENTS for inline parameterization
  • Can reference external files with @path/to/file

BEST FOR

  • Repeatable one-shot tasks: code review, PR description, write unit tests
  • Team workflows: generate a sprint summary, draft a status update, scaffold a new component
  • Short, focused tasks with a clear deliverable
  • Prompts that should be version-controlled alongside the code
SKILLS

SKILLS

STRUCTURE

# YAML frontmatter required
---
name: my-skill
description: What this skill does
---

# Directory layout
.claude/skills/my-skill/
├── SKILL.md       # orchestrator
└── resources/
    ├── topic-a.md
    └── topic-b.md

BEST FOR

  • Multi-step sessions: code audits, project onboarding, system design reviews
  • Tasks that need extra reference files loaded at runtime — e.g., project context, coding standards, client requirements
  • Reusable workflows shared across teams: bug triage, tech debt analysis, migration planning
  • Any flow that needs its own orchestration logic beyond a single prompt
  • Invoked with /skill-name — Claude loads SKILL.md and follows its instructions

BEST PRACTICES

COMMAND AUTHORING RULES

STRUCTURE

  • One command, one job — keep each file under ~50 lines
  • Name files descriptively: pr-description.md, not pr.md
  • Use $ARGUMENTS for any value that varies per invocation
  • Reference large context with @FILEPATH — don't inline it
  • Lead with intent: the first sentence tells Claude what to produce

GOVERNANCE

  • Commit command files to version control — they are team APIs
  • Place team-wide commands in the project repo; personal commands in ~/.claude/commands/
  • Project-level commands override user-level on name collision
  • Review commands in PR just like code — they shape model behavior
BEST PRACTICES

SKILL RESOURCES PATTERN

HOW IT WORKS

  • SKILL.md is the router — it decides which resource to load and when
  • Each file in resources/ covers one concern: one domain, one topic, one workflow stage
  • SKILL.md references resources with relative paths: ./resources/CCA_Domain1.md
  • Claude reads only the file needed for the current step — not all resources at once

WHY IT MATTERS

  • Keeps context window lean — load only what's needed, when it's needed
  • Each resource file can be updated independently without touching the orchestrator
  • Deep content (domain explanations, reference tables) lives in resources, not in the orchestrator
  • Verify all referenced paths exist before shipping — broken references fail silently
BEST PRACTICES

MANAGING THE CONTEXT WINDOW

The problem: every tool call result, resource file, and exchange turn occupies context. As context grows, attention dilutes — critical instructions near the top receive less weight than they should.

LOAD LAZILY

  • Never load all resources upfront
  • Route to the right file based on user intent before reading it
  • Use @file references in commands instead of inlining content

WRITE A MANIFEST

  • In SKILL.md, list available resources with one-line descriptions
  • Claude reads the manifest to pick the right file — not all files
  • Manifest stays in context; bulky resources rotate in and out

USE /COMPACT

  • /compact summarizes the conversation history in place
  • Critical for long skill sessions that span many exchanges
  • Run it before loading a new large resource to free headroom
